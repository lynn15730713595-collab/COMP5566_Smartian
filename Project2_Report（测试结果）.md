# 以太坊智能合约漏洞检测工具实证研究报告

## 项目概述

本项目对四款主流智能合约安全检测工具进行实证研究，评估其在四种不同类型漏洞上的检测能力。研究工具包括 Slither（静态分析）、Mythril（符号执行）、Echidna（模糊测试）和 Smartian（混合方法）。

---

## 1. 实验设置

### 1.1 数据集来源

本实验使用的数据集来自 [SmartBugs Curated](https://github.com/smartbugs/smartbugs-curated)，该数据集包含已人工标注 ground truth 的漏洞合约，可直接运行工具进行测试。

### 1.2 漏洞分类体系

| 类型 | 标注 | SWC 编号 | 描述 | 样本数 |
|:---:|:---:|:---:|:---|:---:|
| T1 | 重入攻击 | SWC-107 | 递归调用导致资产被盗 | 5 |
| T2 | 整数溢出 | SWC-101 | 算术运算超出边界 | 5 |
| T3 | 访问控制 | SWC-105/106 | 权限验证缺失或错误 | 5 |
| T4 | 未检查调用 | SWC-104 | 低级调用返回值未检查 | 5 |
| **合计** | | | | **20** |

### 1.3 评估指标

- **精确率 (Precision)** = TP / (TP + FP)
- **召回率 (Recall)** = TP / (TP + FN)
- **F1 值** = 2 × Precision × Recall / (Precision + Recall)
- **效率指标**: 分析时间、超时率

## 5. 根因分析

### 5.1 抽象层次不匹配

现有工具的设计目标是检测**合约内部的句法模式**（SWC 注册表所定义的漏洞）。

### 5.2 状态空间爆炸

符号执行（Mythril）和模糊测试（Echidna/Smartian）基于状态机运行。随着合约复杂度增加，状态空间呈指数级增长，可能导致分析超时或内存溢出。

在本次实验中，Mythril 的 BECToken.sol 合约因路径爆炸导致分析超时（10% 超时率），正是状态空间爆炸问题的具体体现。

### 5.3 T4 的特殊性：静默失败

未检查低级调用（T4）说明了工具检测方法之间的分化：静态分析可以识别"未检查返回值"这一语法特征，而模糊测试依赖**可观测的错误状态**触发报告。

---

## 6. 结论

1. **T1-T4 可检测**: 传统智能合约漏洞（重入、溢出、访问控制、未检查调用）可被现代工具有效检测
2. **方法论差异**: 静态分析快但浅，符号执行深但慢，模糊测试需手工属性编写
3. **工具推荐**:
   - 快速扫描 → Slither
   - 深度分析 → Mythril
   - 属性验证 → Echidna
   - 综合平衡 → Smartian
